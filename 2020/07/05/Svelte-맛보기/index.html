Svelte 맛보기 🐤

2020/07/05

혜성☄️☄️같이 등장한 새로운 프레임워크가 있다.

공식 홈페이지에서도 느껴지는 자존감 뿜뿜 글에 매력을 느끼게 되었다. 공홈을 읽다보면 뭔가 아래같은 느낌을 많이 느끼게 되는 것 같다.

다 비켜! Svelte 등장이다!!

(주의: 구글 번역기를 이용해 공식 홈페이지의 내용을 해석한 것으로 원뜻과 다를 수 있으니 꼭 공홈에 올라온 자존감 뿜뿜 글들을 읽어야 함)

Svelte가 그래서 뭐지?

Svelte는 React, Vue와 마찬가지로 유저 인터페이스를 쉽게 구축할 수 있도록 등장한 프레임워크이다.

그러나 중요한 차이점이 있으니, 그것은 바로 런타임에 코드를 해석하지 않고 빌드시 Javascript 코드로 변환한다는 것이다. 이 말은 즉, 프레임워크의 추상화에 더이상 시간을 쏟지 않아도 되며 또한 앱이 처음 로드될 때 시간이 절약된다는 말이다!

공식 홈페이지에서는 크게 세가지를 장점으로 내세우고 있다. (더 있는 것 같은데 영어를 못하니 가장 크게 블락 잡혀있는 것 세가지를 이 글에서도 언급하고자 한다.)

Write less code (적은 코드 작성)

Svelte는 다른 프레임워크들과 다르게 적은 코드로 앱을 구성할 수 있다는 것 같다. 실제로 같은 동작을 Svelte와 React로 코딩했을 때, 코드의 양이 확실히 줄어든 것을 확인할 수 있었다.

in svelte

in react

in vue

공식 홈페이지에서는 모든 코드는 버그를 내제하고 있으며, 이로써 코드가 많아질수록 그것은 당신의 앱에 버그가 많아진다는 것을 의미하기도 한다고 말하고 있다.

또한 Svelte에서 컴포넌트는 원하는 만큼 자식 요소를 가질 수 있다. React나 Vue에서는 최상위 요소를 하나 가져야 하는데, 이말은 즉 React를 예로 들면 하나의 컴포넌트에서는 하위에 여러개의 요소를 가질 수 없기에 <> 나 <div> 를 써서 감싸주어야 한다. 그러나 Svelte에서는 이것이 필요 없는 과정인 것이다!

또한 컴포넌트의 상태 선언에 관해서도 굉장히 간단하다. React 의 경우, const [state, setState] = useState('') 등으로 선언하여 사용할 수 있는데, Svelte는 그냥 let state = '' 로 선언해주면 사용할 수 있다.

또한.. React에서 사용되는 useEffect, useMemo, useCallback 등의 동작이 어떠한 보일러플레이트의 활용없이 Svelte 자체에서 수행되고 있다고 한다.

Svelte는 컴파일러이기 때문에 JavaScript의 특성에 국한되지 않습니다. 언어의 의미에 맞게 구성 요소를 작성하는 대신 구성 요소 작성 환경을 디자인 할 수 있습니다.

WOW 😳

No Virtual DOM !!

Virtual DOM is pure overhead
Let’s retire the ‘virtual DOM is fast’ myth once and for all

공식 홈페이지에서 No Virtual DOM 섹션을 클릭하면 상단에 작성되어 있는 문구이다. 과연 그들은 어떻게 Virtual DOM이라는 신화를 없앨 수 있었을까?

Javascript 프레임워크를 공부하거나 사용하면서 많이 들었던 말중에 ‘Virtual DOM은 빠르다’ 가 있을 것이다. 이말은 즉 ‘실제 DOM보다 빠르다’ 라는 말이다.

Svelte는 ‘가상 DOM은 실제 DOM보다 빠르다.’ 라는 말이 밈처럼 사용되고 있다고 말한다.

가상 돔은 실제 DOM에서의 최종 조작에 추가된다고 한다. 즉 가상 DOM을 쓴다고해서 빠르다고 하기 어렵다는 것..?

그럼 오버헤드는 어디서 발생하는 걸까?

먼저 새로운 가상 DOM을 이전 스냅 샷과 비교하지 않으면 실제 DOM에 변경 사항을 적용 할 수 없다.

만약 name이라는 prop값을 everybody로 변경한다고 해보자.

  1. 두 스냅 샷 모두 단일 요소를 포함한다. 두 경우 모두 <div>이므로 동일한 DOM 노드를 유지할 수 있다.
  2. 이전 <div> 및 새 <div>의 모든 속성을 열거하여 변경, 추가 또는 제거해야하는지 확인한다. 두 경우 모두 “greeting”값을 가진 className이라는 단일 속성이 있다.
  3. 요소로 내려 가면 텍스트가 변경되었음을 알 수 있으므로 실제 DOM을 업데이트 해야한다.

이 세 단계 중 대부분은 업데이트의 경우와 마찬가지로 앱의 기본 구조가 변경되지 않았으므로이 경우에는 3단계만 가치가 있다. 3단계로 바로 건너 뛸 수 있다면 훨씬 더 효율적일 것이다.

기존 UI 프레임워크와 달리 Svelte는 위 업데이트 코드를 생성하여 런타임에 작업을 기다리지 않고 빌드 시간에 앱에서 변경 사항이 어떻게 발생하는지 알고 있는 컴파일러이다.

그럼 프레임워크들은 가상 돔을 왜 쓸까?

가상 DOM은 기능이 아니라는 것을 이해해야 한다. 이는 목적을위한 수단이며, 목적은 선언적이고 상태 중심의 UI 개발을 의미한다. Virtual DOM은 일반적으로 충분한 성능으로 상태 전이에 대해 생각하지 않고도 앱을 빌드 할 수 있기 때문에 유용하다. 이는 버그가 적고 지루한 작업 대신 창의적인 작업에 더 많은 시간을 소비한다는 것을 의미한다.

그러나 가상 DOM을 사용하지 않고도 유사한 프로그래밍 모델을 얻을 수 있다는 것이 밝혀졌다. 바로 여기, Svelte가 왔다.

Truly reactive

Svelte는 3버전으로 업데이트 되면서 반응성 부분에서 업데이트가 생긴 것 같다.

이전 Svelte에서는 this.set 메소드를 호출하여 일부 상태가 변경되었음을 컴퓨터에 알려줬다.

hooks의 등장으로 많은 것이 변화하기 시작했는데, 많은 프레임 워크가 자체적인 hook 구현을 실험하기 시작했지만 Svelte는 본인들이 원하는 방향이 아니라는 결론을 신속하게 내렸다.

중요한 것은 프록시 나 접근자를 사용하는 오버 헤드와 복잡성없이이 모든 것을 할 수 있다는 것이다. 저것은 변수일 뿐이다.
